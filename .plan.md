# Plan: Lock Down Port Exposure + Harden Default Credentials

## Problem

1. All Docker ports (8087, 8089, 19023, 8443) bind to `0.0.0.0` — reachable from any interface, not just Tailscale.
2. `serve_packages()` HTTP server binds to `0.0.0.0`.
3. Native-mode FTSConfig sets `FTS_DP_ADDRESS` and `FTS_USER_ADDRESS` to `0.0.0.0`.
4. System-wide default credentials are `team` / `team` (FTS_PASSWORD = FTS_USERNAME).

## Changes (5 files + 1 todo cleanup)

---

### Change 1: `docker/docker-compose.yml` — bind ports to specific IPs

**What**: Prefix each port mapping with a host IP so Docker only listens on that interface.

**Exact edit** — replace the `ports:` block (lines 8-12):

```yaml
# BEFORE
ports:
  - "${COT_PORT:-8087}:${COT_PORT:-8087}"
  - "${SSL_COT_PORT:-8089}:${SSL_COT_PORT:-8089}"
  - "${API_PORT:-19023}:${API_PORT:-19023}"
  - "${DATAPACKAGE_PORT:-8443}:8443"
```

```yaml
# AFTER
ports:
  - "${SERVER_IP:-127.0.0.1}:${COT_PORT:-8087}:${COT_PORT:-8087}"
  - "${SERVER_IP:-127.0.0.1}:${SSL_COT_PORT:-8089}:${SSL_COT_PORT:-8089}"
  - "127.0.0.1:${API_PORT:-19023}:${API_PORT:-19023}"
  - "${SERVER_IP:-127.0.0.1}:${DATAPACKAGE_PORT:-8443}:8443"
```

**Why each line**:
- CoT/SSL/DataPackage → `${SERVER_IP}` so TAK clients on the VPN can reach them, but nothing outside that interface can.
- API → hardcoded `127.0.0.1` because all API calls use `docker exec` (they run inside the container). The API port never needs to be reachable from outside the host.
- Fallback `127.0.0.1` if SERVER_IP is unset — safe default (nothing exposed).

**Precondition**: `SERVER_IP` is already exported in `_docker_start()` at `lib/server.sh:40`. No additional export needed.

**Verification**: After `./heartbeat start`, run `ss -tlnp | grep -E '8087|8089|19023|8443'`. Each port should show the Tailscale IP (e.g. `100.85.89.14`) or `127.0.0.1`, never `0.0.0.0` or `*`.

### Change 2: `lib/package.sh` — bind `serve_packages()` to SERVER_IP

**What**: Change the `python3 -m http.server` bind address from `0.0.0.0` to `${SERVER_IP}`.

**Exact edit** — line 270:

```bash
# BEFORE
python3 -m http.server "$port" --bind 0.0.0.0 2>/dev/null

# AFTER
python3 -m http.server "$port" --bind "${SERVER_IP}" 2>/dev/null
```

**Why**: The download page should only be reachable from the VPN, same as all other services.

**Verification**: After `./heartbeat serve`, run `ss -tlnp | grep 9000`. Should show `100.85.89.14:9000`, not `0.0.0.0:9000`.

### Change 3: `lib/install.sh` — lock down native-mode FTSConfig

**What**: In `install_native_mode()`, change FTS_DP_ADDRESS/FTS_USER_ADDRESS from `0.0.0.0` to `${SERVER_IP}`, and FTS_API_ADDRESS from `0.0.0.0` to `127.0.0.1`.

**Exact edit** — inside the native-mode YAML heredoc (lines 148-152):

```yaml
# BEFORE
  FTS_DP_ADDRESS: "0.0.0.0"
  FTS_USER_ADDRESS: "0.0.0.0"
  FTS_API_PORT: ${API_PORT}
  FTS_FED_PORT: 9000
  FTS_API_ADDRESS: "0.0.0.0"

# AFTER
  FTS_DP_ADDRESS: "${SERVER_IP}"
  FTS_USER_ADDRESS: "${SERVER_IP}"
  FTS_API_PORT: ${API_PORT}
  FTS_FED_PORT: 9000
  FTS_API_ADDRESS: "127.0.0.1"
```

**Why**: Docker mode already does this correctly (lines 71-75 use `${SERVER_IP}` for DP/USER). Native mode was left at `0.0.0.0`, meaning FTS would bind to all interfaces AND data packages would embed `0.0.0.0` as the server address (which wouldn't even work for clients). `FTS_API_ADDRESS` locked to loopback since the API should never be network-accessible.

**Verification**: After native-mode setup, inspect `data/fts/FTSConfig.yaml` — confirm no `0.0.0.0` entries.

### Change 4: `lib/common.sh` — add `gen_password()` helper

**What**: Add a new function after `gen_secret()` (line 155).

```bash
gen_password() {
    head -c 12 /dev/urandom | base64 | tr -d '/+=' | head -c 12
}
```

**Why**: 12 alphanumeric chars = ~71 bits of entropy. Short enough to type/read from a screen, strong enough that it can't be guessed. Mirrors the pattern of `gen_secret()` which already exists.

**Verification**: Run `source lib/common.sh && gen_password` — should output a 12-char alphanumeric string. Run it twice — outputs should differ.

### Change 5: `setup.sh` — generate random default password

**What**: Change the default FTS_PASSWORD from `${fts_user}` to a random value.

**Exact edit** — lines 259-265:

```bash
# BEFORE
    # ---- Default credentials ----
    local fts_user="team"
    local fts_pass="${fts_user}"
    if $INTERACTIVE; then
        echo ""
        fts_user=$(prompt_default "Default TAK username" "$fts_user")
        fts_pass=$(prompt_default "Default TAK password" "$fts_user")
    fi

# AFTER
    # ---- Default credentials ----
    local fts_user="team"
    local fts_pass
    fts_pass=$(gen_password)
    if $INTERACTIVE; then
        echo ""
        fts_user=$(prompt_default "Default TAK username" "$fts_user")
        fts_pass=$(prompt_default "Default TAK password" "$fts_pass")
    fi
```

**Why**: Eliminates `password == username` default. Non-interactive mode gets a random password automatically. Interactive mode shows the generated password as the default (user can accept or override).

**Note on per-user package passwords**: `_generate_fts_package()` still uses `password = safe_name` (lib/package.sh:42). This is acceptable per the todo's own guidance: "keep name=name only for private/VPN deployments." After Change 1 locks ports to the VPN interface, all deployments are effectively VPN-only. The SSL certs in the package are the real authentication mechanism.

**Verification**: Run `./setup.sh` (non-interactive). Inspect `config/heartbeat.conf` — `FTS_PASSWORD` should be a 12-char random string, not "team".

## What is NOT changed (and why)

- **`_generate_fts_package` per-user password** (`password = safe_name`): Kept as-is. Changing to random would break re-generation (FTS API returns "exists" on second run, so the new random password wouldn't match the stored one, and the displayed credentials would be wrong). The todo explicitly allows name=name for VPN deployments.
- **Docker-mode `FTS_API_ADDRESS: "0.0.0.0"`** (install.sh:75): This is the bind address *inside* the container. Docker port bindings (Change 1) control external access. Changing this to 127.0.0.1 inside the container would break `docker exec` API calls that connect to `127.0.0.1:$port` — those connect to the container's loopback, which is the same as 0.0.0.0 inside the container. So this is correct as-is.
- **Tailscale ACLs**: Out of scope. Binding to the Tailscale interface is sufficient network-level restriction. Tailscale ACLs are an additional layer users can configure themselves.
